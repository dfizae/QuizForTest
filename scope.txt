# [출제 범위 1. 릴레이션 용어 (튜플/도메인/카디널리티)]

## 📝 문제 (Problem)

릴레이션 용어 설명에서 (①~③)에 들어갈 답을 `<보기>`에서 찾아 쓰시오. (5점)

- (①): 릴레이션의 각 행(파일 구조에서는 레코드)
- (②): 속성들의 데이터 타입이 정의되어 구체적인 값을 가짐
- (③): 튜플의 개수

`<보기>` 도메인, 차수, 속성, 튜플, 디그리, 카디널리티, 릴레이션 스키마, 릴레이션 인스턴스

## 🔍 상세 분석 (Analysis)

- (①) 행 = 튜플
- (②) 속성들의 데이터 타입이 정의되어 구체적인 값을 가지는 실제 테이블(튜플들의 집합) = 릴레이션 인스턴스
- (③) 튜플 개수 = 카디널리티

## 🎓 학생 가이드 (암기/풀이 요령)

- 한 줄 요약: 행=튜플, 값의 범위=도메인, 행의 개수=카디널리티
- 실수 포인트: 차수/디그리(속성 수)랑 카디널리티(튜플 수) 헷갈리지 않기

# [출제 범위 2. 디자인 패턴]

## 생성(Creational)

- Singleton
    
    인스턴스를 1개만 생성/공유하도록 보장. 키워드: 전역 1개, 공유 객체
    
- Factory Method
    
    객체 생성 코드를 서브클래스(또는 구현체)로 위임해, 생성 로직을 캡슐화. 키워드: 생성 책임 위임, 다형성 생성
    
- Abstract Factory
    
    관련 있는 객체 “묶음(제품군)”을 생성하는 인터페이스 제공. 키워드: 제품군, 팩토리의 팩토리
    
- Builder
    
    복잡한 객체를 단계적으로 조립해서 생성(생성과 표현 분리). 키워드: 단계적 생성, 조립
    
- Prototype
    
    기존 객체를 복사(clone)해서 새 객체 생성. 키워드: 복제, clone
    

## 구조(Structural)

- Adapter
    
    호환 안 되는 인터페이스를 변환해서 연결. 키워드: 변환, 호환
    
- Bridge
    
    추상(Abstraction)과 구현(Implementor)을 분리해서 각각 독립적으로 확장. 키워드: 추상-구현 분리, 독립 확장
    
- Composite
    
    객체들을 트리 구조로 구성해, 단일 객체/복합 객체를 동일하게 다룸. 키워드: 트리, 부분-전체, 동일 취급
    
- Decorator
    
    객체에 기능을 동적으로 추가(상속 대신 조합으로 확장). 키워드: 기능 추가, 래핑, 동적 확장
    
- Facade
    
    복잡한 서브시스템을 감추고 단순한 통합 인터페이스 제공. 키워드: 단일 창구, 단순화
    
- Proxy
    
    대리 객체가 실제 객체 접근을 제어/대신 수행. 키워드: 대리, 접근제어, 지연로딩/캐싱
    

## 행위(Behavioral)

- Chain of Responsibility (책임 연쇄)
    
    요청을 여러 핸들러가 순서대로 넘기며 처리(처리자 결합도 감소). 키워드: 체인 전달
    
- Command
    
    요청을 객체로 캡슐화(실행/취소/로그 등 가능). 키워드: 요청=객체, undo 가능
    
- Interpreter
    
    문법(규칙)을 클래스로 표현해 해석/평가. 키워드: 문법 규칙, 해석기
    
- Iterator (반복자)
    
    내부 구조를 노출하지 않고 순차 접근 방법 제공. 키워드: 순회, 접근 캡슐화
    
- Mediator
    
    객체 간 직접 참조를 줄이고 중재자가 상호작용을 통제. 키워드: 중재, 결합도 감소
    
- Memento
    
    객체 상태를 저장했다가 복원(스냅샷/undo). 키워드: 상태 저장/복구
    
- Observer (옵서버)
    
    한 객체 상태 변화 시, 등록된 다수 객체에 자동 통지/갱신. 키워드: 발행-구독, 알림
    
- Strategy
    
    알고리즘을 교체 가능하게 캡슐화(런타임 교체). 키워드: 알고리즘 교체
    
- Template Method
    
    알고리즘 뼈대는 상위 클래스에, 일부 단계는 하위 클래스가 구현. 키워드: 골격 고정, 일부 재정의

# [출제 범위 4. Ad-hoc Network]

## 1) Infrastructure Network (인프라스트럭처)

- 한 줄 정의: AP/기지국/스위치 같은 **중앙 인프라(기반 시설)**가 있고, 단말은 거기에 **접속해서 통신**
- 핵심 키워드: “기지국(AP) 있음”, “중앙 장비 통해 통신”
- 예시: 회사 Wi‑Fi(공유기/AP에 붙음), 일반적인 휴대폰 통신(기지국 기반)
- 헷갈림 방지: Ad-hoc이랑 대비로 외움 → 인프라 있으면 Infrastructure

## 2) Ad-hoc Network (애드혹)

- 한 줄 정의: AP/기지국 없이 **단말끼리 직접(임시로) 네트워크 구성** (필요하면 멀티홉)
- 핵심 키워드: “인프라 없음”, “임시/자율 구성”, “(가능하면) 멀티홉 라우팅”
- 예시: 재난/군사 현장 임시 통신망, 단말들끼리 즉석 연결
- 헷갈림 방지: “그때그때(ad-hoc) 바로 만든다” = 인프라 없이 즉석

## 3) Mesh Network (메시)

- 한 줄 정의: 노드들이 **그물망처럼 여러 경로로 연결**되어, 한 경로가 끊겨도 **다른 경로로 우회(자기복구)** 가능
- 핵심 키워드: “다중 경로”, “릴레이/중계”, “장애에 강함(우회)”
- 예시: 메시 Wi‑Fi(집에 공유기 여러 대), 재난 통신망
- Ad-hoc과 관계:
    - Ad-hoc은 “인프라 없이 즉석”이라는 **구성 방식**
    - Mesh는 “여러 노드가 서로 중계하며 다중 경로”라는 **토폴로지(연결 구조)**
    - 즉, Ad-hoc 네트워크가 Mesh 형태를 띨 수도 있음(겹칠 수 있음)

## 4) Peer-to-Peer Network (P2P, 피어투피어)

- 한 줄 정의: 서버 중심이 아니라, 참여한 컴퓨터(피어)들이 **서로 대등하게** 자원을 주고받는 방식
- 핵심 키워드: “서버 없음/약함”, “각자 클라이언트이자 서버”, “자원 공유”
- 예시: 토렌트 같은 파일 공유, 일부 메신저/분산 서비스
- 헷갈림 방지: Mesh(연결 구조)랑 다름
    - Mesh는 “누가 누구랑 연결돼 있나(다중 경로)”
    - P2P는 “역할이 대등한가(서버-클라 구조냐)”

## 5) Virtual Private Network (VPN)

- 한 줄 정의: 공중망(인터넷) 위에 **암호화된 터널**을 만들어 **사설망처럼** 안전하게 접속
- 핵심 키워드: “터널링”, “암호화”, “원격지에서 내부망 접속”
- 예시: 집에서 회사 내부망 접속, IPsec/SSL VPN
- 헷갈림 방지: 이건 “네트워크 구조”가 아니라 **보안/접속 기술**임 (Mesh/P2P/Ad-hoc 같은 급이 아님)

## 6) Sensor Network (센서 네트워크)

- 한 줄 정의: 온도/습도/조도 같은 **센서 노드**들이 데이터를 모아 **게이트웨이(싱크 노드)**로 전달하는 네트워크(주로 IoT)
- 핵심 키워드: “센서/IoT”, “저전력”, “데이터 수집 중심”
- 예시: 스마트팜, 공장 설비 모니터링, 환경 감시
- 관계: 구현은 스타(중앙 수집), 트리, 또는 Mesh로도 가능 → “목적/구성원(센서)”로 구분하는 게 포인트

---

## 시험용 10초 구분 체크(빠른 판별)

- AP/기지국 같은 **중앙 장비 있나?** → 있으면 Infrastructure / 없으면 Ad-hoc 후보
- **여러 노드가 중계해서 우회 경로** 만들나? → Mesh
- **서버 없이 대등하게 파일/자원 공유**하나? → P2P
- **인터넷 위 암호화 터널로 사설망 접속**이 목적? → VPN
- **센서들이 데이터 수집**하는 IoT 망? → Sensor Network

## 🏗️ 체크리스트

- 재난/군사 + 임시 + 모바일만 + 멀티 홉

# [출제 범위 5.  정보 보안 프로토콜]

SSH
(Secure SHell)
. 서로 연결된 컴퓨터 간 원격 명령실행 및 셀 서비스 수행
. 키를 통한 인증은 클라이언트의 공개키를 서버에 등록해야 함
. 전송 데이터는 암호화되며, 기본적으로 22번 포트 사용

SSL
(Secure Socket Layer)
. 웹 브라우저 ↔ 서버 간 안전한 데이터 전송 목적
. https로 시작 / 443번 포트 사용

TLS
(Transport Layer
Security)
. 인터넷 커뮤니케이션을 위한 개인 정보와 데이터 무결성을
제공하는 보안 프로토콜 / SSL의 개선 버전

IPSec
. IP 네트워크 계층을 안전하게 보호하기 위한 보안 프로토콜
→ 전송 모드 (트래픽 경로 유출 가능) / 터널 모드 (패킷 전체)
. AH(Authentication Header)와 ESP(Encapsulating Security
Payload)라는 두 가지 보안 프로토콜을 사용

S-HTTP . 웹상에서 네트워크 트래픽을 암호화 / 기존 HTTP의 보안 확장

TKIP
(Temporal Key
Integrity Protocol)
. 임시 키 무결성 프로토콜 / WEP 취약성을 보완하기 위해
암호 알고리즘의 입력 키 길이를 128비트로 늘림

# [출제 범위 6.  서비스 공격 유형]

## 1) DoS / DDoS 기본

- DoS: 단일 공격으로 서비스 마비
- DDoS: 분산(다수) 공격으로 서비스 마비
- Botnet(좀비 PC): DDoS 수행 주체

## 2) 스푸핑(Spoofing, 위조)

- IP Spoofing: 출발지 IP 위조(증폭/반사 공격 핵심)
- ARP Spoofing(ARP Poisoning): 같은 LAN에서 MAC 속여 트래픽 가로채기
- DNS Spoofing(DNS Cache Poisoning): DNS 응답/캐시 위조

## 3) 대표 DoS/DDoS 공격 “이름들”

- SYN Flooding: TCP 연결(3-way) 자원 고갈
- UDP Flooding: UDP 대량 전송으로 대역폭/자원 고갈
- ICMP Flooding(Ping Flood): ICMP 대량 전송
- Smurf 공격: ICMP + 브로드캐스트 + IP 스푸핑 → 응답 증폭
- Ping of Death: 비정상적으로 큰 Ping(구형 개념이지만 용어로 출제)
- TearDrop: IP 단편화(fragment) 재조립 교란(구형 개념이지만 용어로 출제)
- LAND Attack: 출발지=목적지 등 비정상 패킷(구형 개념이지만 용어로 출제)

# [출제 범위 7.  정보 보안 침해 공격]

■ URL 구조
스킴://사용자이름:비밀번호@호스트:포트/경로?질의#프레그먼트

- 스킴 (Scheme)
사용할 프로토콜 명시 (리소스 요청 및 접근 방법)
ex. http, https, ftp, mailto(이메일), rtsp(스트리밍) 등
사용자 정보 User info : 사용자 이름 + 비밀번호

- 호스트 (Host) 도메인명 또는 IP 주소

- 포트 (Port)
Socket(소켓) 구분 역할
생략 시 http는 80, https는 443로 기본값 포트번호 설정

- 경로 (Path) 호스트에서 제공하는 자원의 경로 (계층적 구조)

- 질의 (Query)
Query parameter 또는 Query string으로 불림
key=value 형태, &로 질의 추가 (ex. ?directoryNo=0&currentPage=1)

- 프레그먼트 (Fragment)
내부 북마크 기능 (웹문서 해시태그) / 해시태그로 이동을 원하는 요소
의 id를 링크로 연결 시 스크롤 이동없이 해당 위치로 바로 이동 가능
